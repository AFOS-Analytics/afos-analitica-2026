0
Auditoria de Segurança
~3h
Dia 1 (manhã)
Por que isso importa: Quando o repo ficar público, qualquer pessoa no mundo poderá ler cada linha de código e cada commit do histórico. Se houver uma API key, token ou senha em qualquer lugar — mesmo num commit antigo que já foi "apagado" — alguém vai encontrar. Isso acontece em minutos, não dias. Bots varrem repos novos automaticamente buscando secrets.
#TarefaTempoQuem
0.1
Varrer secrets em todos os arquivos
Buscar API keys, tokens, senhas hardcoded. Inclui o token Vercel que está no skill /atualizar.
 1h Claude
0.2
Varrer histórico git
Verificar se algum commit antigo contém secret no diff. Se sim, será necessário reescrever o histórico com git filter-branch.
 45min Claude
0.3
Validar .env.example
Confirmar que todas as variáveis necessárias estão listadas, sem valores reais. Quem clonar o repo precisa saber o que configurar.
 20min Claude
0.4
Validar .gitignore
Confirmar que .env.local, node_modules, .next, e qualquer arquivo sensível estão sendo ignorados pelo git.
 10min Claude
0.5
Remover dados pessoais do código
Verificar se seu email pessoal, IP ou dados sensíveis aparecem em algum arquivo ou comentário.
 30min Claude
0.6
Revisar resultado juntos
Você valida o que foi encontrado e decide como resolver cada item.
 15min André
Dica: Essa fase é a mais importante. Todo o resto pode ser ajustado depois do lançamento. Um secret vazado não pode ser "desvazado".
1
Preparação do Repositório
~6h
Dias 1-2
Por que isso importa: Um desenvolvedor leva 30 segundos para decidir se um repo é sério ou amador. Eles olham: tem CONTRIBUTING? Tem issues abertas? Tem templates? Tem lint? Projetos sem isso passam a impressão de "projeto pessoal jogado no GitHub". Projetos com isso passam a impressão de "isso aqui é profissional, posso contribuir". Você quer a segunda impressão.
#TarefaTempoQuem
1.1
CONTRIBUTING.md
Documento que explica: como rodar o projeto local, padrões de código, como abrir PR, como reportar bug. É o "manual do contribuidor".
 1.5h Claude
1.2
CODE_OF_CONDUCT.md
Código de conduta da comunidade. Usa o modelo "Contributor Covenant" (padrão da indústria). Diz que a comunidade é respeitosa e inclusiva.
 20min Claude
1.3
SECURITY.md
Como reportar vulnerabilidades de segurança de forma privada (email), em vez de abrir issue pública. Padrão em projetos sérios.
 15min Claude
1.4
Issue templates (Bug + Feature)
Formulários padronizados para quando alguém abre uma issue. Bug report pede: o que aconteceu, o que esperava, como reproduzir. Feature request pede: o que quer, por quê.
 30min Claude
1.5
PR template
Checklist que aparece quando alguém abre um Pull Request. Ex: "Testei localmente? Atualizei documentação? Lint passou?"
 15min Claude
1.6
Labels no GitHub
Etiquetas coloridas: "good first issue" (verde, para iniciantes), "help wanted" (amarelo), "enhancement", "bug", "documentation". Organiza as issues.
 15min André
1.7
Criar 5-10 issues iniciais
Issues com "good first issue" que convidam devs a contribuir. Ex: "Add country: Mexico", "Translate to French", "Add PredictIt as data source". Mostra que o projeto está vivo e aberto.
 1h Juntos
1.8
Configurar ESLint
Ferramenta que verifica qualidade do código automaticamente. Quem clonar e rodar "npm run lint" precisa funcionar. Hoje está pendente de configuração.
 1h Claude
1.9
Descrição + Topics no GitHub
A frase curta que aparece abaixo do nome do repo no GitHub + tags como "prediction-markets", "elections", "polymarket", "open-source", "next-js". Ajuda na busca.
 15min André
1.10
Social Preview Image
Imagem 1280x640px que aparece quando alguém compartilha o link do repo no X, Slack, WhatsApp. Precisa ser atraente — é a "capa" do projeto.
 1h Juntos
2
Assets Visuais
~10h
Dias 3-5
Por que isso importa: 80% das pessoas que vão ver o seu projeto NÃO vão clicar no link do site. Elas vão olhar o README no GitHub e decidir em segundos se é interessante. Se não tiver imagens, GIF ou vídeo — elas passam direto. Um screenshot bem feito do dashboard com dados reais vale mais que 10 parágrafos de texto.
#TarefaTempoQuem
2.1
Screenshot do Dashboard
Captura de tela do dashboard com dados reais visíveis — Polymarket, pesquisas, candidatos. Resolução alta, full width. Tirar no Chrome com DevTools aberto para simular tela grande.
 1h André
2.2
Screenshot do Mapa Global
Mapa D3.js mostrando países coloridos, com tooltip visível em um país (ex: Brasil). Mostra que é multi-país.
 30min André
2.3
Screenshot da Landing Page
Landing page nos dois temas (azul/branco) mostrando o visual profissional.
 30min André
2.4
GIF animado (15-20 segundos)
Gravação rápida navegando: landing → dashboard → scroll nas seções → mapa global → click num país. Usar ferramenta como ScreenToGif (Windows, grátis) ou LICEcap. O GIF vai no topo do README.
 2.5h André
2.5
Vídeo demo (60-90 segundos)
Screencast com narração em inglês mostrando o produto. Não precisa ser profissional — pode ser gravado com OBS Studio (grátis) + microfone do notebook. O importante é mostrar o produto funcionando e explicar o diferencial em 1 minuto. Vai para o Product Hunt e YouTube.
 4h André
2.6
Inserir assets no README (EN + PT-BR)
Adicionar screenshots e GIF nos dois READMEs, logo após o header e antes da seção "About". As imagens ficam em public/ ou num serviço como imgur (para não pesar o repo).
 1h Claude
Dica para iniciante: O vídeo é a tarefa mais demorada. Não busque perfeição — um vídeo "bom o suficiente" gravado em 1 take é melhor que um vídeo "perfeito" que nunca sai. Escreva um roteiro de 8-10 frases antes de gravar. Pratique 2x. Grave na 3ª.
3
Textos de Lançamento
~10h
Dias 5-8
Por que isso importa: Cada plataforma tem um público e um tom diferentes. O que funciona no Hacker News (técnico, sem hype) seria ignorado no Product Hunt (visual, storytelling). E vice-versa. Preparar os textos com antecedência evita escrever apressado no dia do lançamento — quando o foco precisa ser responder comentários, não redigir posts.
#TarefaTempoQuem
3.1
Título + texto Hacker News
Tom: Técnico, factual, zero hype. Formato: "Show HN: [título ~80 chars]" + primeiro comentário com 10-15 linhas explicando: o que é, por que fiz, stack técnico, link do repo e demo. HN odeia autopromoção — o texto precisa soar como "fiz isso, achei que vocês gostariam de ver".
 2h Juntos
3.2
Product Hunt page
Tom: Visual, polido, amigável. Precisa de: tagline (1 frase), descrição (2-3 parágrafos), 5 screenshots em sequência, vídeo demo, links, categorias. Você cria conta como "Maker" e agenda o launch. A comunidade PH vota durante 24h.
 2.5h Juntos
3.3
Thread X/Twitter (Inglês)
Tom: Storytelling técnico. Formato: 5-7 tweets encadeados. Tweet 1: gancho + screenshot. Tweets 2-5: o que faz, diferencial, stack, dados. Tweet final: link repo + demo. Cada tweet com imagem ou GIF.
 1.5h Juntos
3.4
Thread X/Twitter (Português)
Versão brasileira da thread. Não é tradução literal — adaptar para o contexto e referências BR. Marcar influenciadores brasileiros de tech.
 1h Juntos
3.5
Posts Reddit (3-4 subreddits)
r/dataisbeautiful: foco no mapa + visualização. r/opensource: foco no projeto. r/webdev: foco no stack técnico. r/Polymarket: foco no uso dos dados deles. Cada sub tem regras — ler antes de postar.
 1.5h Claude
3.6
Email de cold outreach (template)
3 linhas máximo. Sem pitch, sem pedir nada. Só mostrar. "We built X. Live at Y. Code at Z. Thought you'd find it relevant." Um template que funciona para Polymarket, Kalshi, jornalistas, newsletters.
 30min Claude
3.7
One-pager / Press Release
1 página PDF: o que é o AFOS, diferencial (primeiro do mundo), números (14 países, 400+ fontes), stack, link. Para enviar a jornalistas que peçam mais info. Visual limpo, marca AFOS.
 1.5h Juntos
4
Lista de Contatos
~4h
Dias 8-9
Por que isso importa: Cold outreach funciona quando é personalizado e direcionado. Enviar email genérico para "info@empresa.com" não funciona. Você precisa encontrar a pessoa certa (dev relations, editor, fundador) e explicar em 3 linhas por que isso importa para ELA especificamente. A lista precisa estar pronta antes do Dia D — no dia do lançamento você só aperta "enviar".
#TarefaTempoQuem
4.1
Contatos Polymarket + Kalshi + Verso
Encontrar emails/X de: dev relations, partnerships, fundadores. Polymarket (Shayne Coplan), Kalshi (Tarek Mansour), Verso (founders). 2-3 contatos por empresa.
 1h Juntos
4.2
Contatos jornalistas tech
TechCrunch, The Verge, Wired, Rest of World (LATAM), Ars Technica. Buscar quem cobre: open source, prediction markets, ou LATAM tech. 5-10 jornalistas com email.
 1h Juntos
4.3
Contatos newsletters
TLDR (tldr.tech), Hacker Newsletter, Console.dev, Changelog, This Week in React, bytes.dev. Cada uma tem formulário de submissão ou email de editor.
 45min Claude
4.4
Contatos acadêmicos + influenciadores
Philip Tetlock (superforecasting), Nate Silver (X), Andrew Gelman (blog). BR: Filipe Deschamps, Akita, Fabio Akita. Contato via X ou email público.
 45min Juntos
4.5
Organizar em planilha
Nome, empresa, canal (email/X/LinkedIn), status (não enviado/enviado/respondeu). Para acompanhar no dia D e depois.
 30min André
5
Pré-Launch (Testes e Agendamento)
~5h
Dias 9-11
Por que isso importa: O dia do lançamento precisa ser 100% execução, 0% improvisação. Tudo que pode ser testado e agendado antes, deve ser. Se o site cair durante o pico de tráfego do HN, você perde a oportunidade — não tem como "relançar". Se o git clone não funcionar do zero, os devs vão embora em 30 segundos.
#TarefaTempoQuem
5.1
Testar git clone + setup do zero
Num computador ou pasta limpa, seguir o README passo a passo: git clone, npm install, configurar .env, prisma migrate, seed, npm run dev. Se qualquer passo falhar, corrigir antes de lançar. Esse teste é OBRIGATÓRIO.
 1.5h Juntos
5.2
Stress test do site
Verificar que afos-analytics.com está estável, cron rodando, dados frescos. Vercel aguenta tráfego bem (CDN global), mas confirmar que Redis e Neon não vão travar com pico de requests.
 1h Claude
5.3
Agendar Product Hunt
Criar conta de Maker no Product Hunt. Montar a page com screenshots, tagline, descrição, vídeo. Agendar para o dia D à meia-noite Pacific Time (PH roda em ciclos de 24h PT).
 1h André
5.4
Revisar todos os textos
Reler HN post, threads X, Reddit, emails. Idealmente pedir para alguém nativo em inglês revisar. Se não tiver, usar o próprio Claude para checagem final.
 1h Juntos
5.5
Definir data de lançamento
Melhores dias: terça, quarta ou quinta. Pior dia: sexta/sábado (menos gente online). Horário: 9am Eastern Time (10am BRT). HN e PH têm pico entre 9-11am ET.
 15min André
5.6
Preparar FAQ mental
Perguntas que vão fazer: "How accurate is Polymarket?", "Why not PredictIt/Kalshi?", "Why Apache 2.0?", "How do I add my country?", "What's next?". Ter respostas prontas para copiar e colar.
 30min Juntos
Dica: O teste de "git clone do zero" (5.1) é o mais importante desta fase. Se um dev clona e não funciona, ele abre uma issue negativa ou simplesmente vai embora. Nenhum texto bonito compensa setup quebrado.
6
Dia do Lançamento
~8h
Dia D
Por que isso importa: O lançamento é um evento de 24 horas. Os algoritmos do HN e Product Hunt valorizam velocidade de votos nas primeiras horas. Se você posta e desaparece, perde. Se você posta e responde todos os comentários, o post sobe. Esse dia precisa ser 100% dedicado — sem reuniões, sem distrações.

7
Pós-Launch (D+1 a D+7)
~6h
Semana seguinte
Por que isso importa: O lançamento abre a porta. A semana seguinte é onde você constrói a comunidade. Devs que deram star estão observando — se você responde issues rápido e aceita PRs, eles voltam. Se o repo fica parado, eles esquecem. "Building in public" (compartilhar métricas e progresso) mantém o momentum.
#TarefaTempoQuem
7.1
Responder todas as issues e PRs
Cada issue merece resposta em menos de 24h. Mesmo que seja "Thanks for the suggestion, we'll look into it." Mostra que o projeto está vivo.
 1h/dia André
7.2
Follow-up com contatos
Quem não respondeu ao cold email no dia D → reenviar em D+3 com update ("launched yesterday, 500 stars in 24h"). Quem respondeu → continuar conversa.
 1h André
7.3
Postar métricas ("building in public")
Tweet/post em D+3 e D+7: "Day 3: X stars, Y forks, contributors from Z countries." Transparência atrai mais contribuidores e press.
 1h André
7.4
Blog post / artigo
"How we built the first platform to cross-reference prediction markets with election polls." Publicar no dev.to, Medium ou blog pessoal. Inclui a história, stack, decisões técnicas, e métricas do lançamento.
 3h Juntos
7.5
Avaliar próximos passos
Com base no feedback recebido: o que a comunidade quer? Quais países pediram? Quais features? Priorizar e criar issues.
 1h Juntos
7.6
Integrar Kalshi — 2º mercado de previsão (D+5 a D+7)
Tacada estratégica pós-launch: Adicionar Kalshi (CFTC-regulated, mercado BR: kxbrpres-26) como segunda fonte de odds ao lado do Polymarket. API pública sem autenticação (api.elections.kalshi.com/trade-api/v2). Replicar padrão existente: lib/kalshi/client.ts + normalize.ts, mesmo cron, Redis key kalshi:*. Dashboard mostra Polymarket vs Kalshi vs Pesquisas lado a lado. Por que agora: Gera segunda onda de visibilidade ("just added Kalshi support!") + dá motivo para Kalshi divulgar o AFOS + valida o posicionamento "5 grandes mercados globais" do README. Anunciar no X/HN como update. ~4-6h de trabalho.
 4-6h Juntos
Por que Kalshi em D+5 e não antes: No dia do lançamento, Polymarket sozinho já é inédito. Guardar Kalshi para a semana seguinte cria uma segunda notícia — mantém o projeto nos feeds e dá motivo para quem já deu star voltar e olhar de novo. É combustível de momentum, não de lançamento.

Cronograma Dia a Dia

Ritmo: 4-5 horas por dia (realista para quem está aprendendo e trabalhando sozinho)

Dia 1

Auditoria de Segurança + Início Repo

Fase 0 completa (varredura de secrets, .env, .gitignore) + iniciar CONTRIBUTING.md e CODE_OF_CONDUCT.md

~5h
Dia 2

Finalizar Preparação do Repo

SECURITY.md, templates de issue/PR, ESLint, criar issues iniciais com "good first issue"

~4h
Dia 3

Screenshots + Início GIF

Capturar screenshots (dashboard, mapa, landing). Instalar ScreenToGif. Começar a gravar o GIF animado.

~4h
Dia 4

GIF + Social Preview + Início Vídeo

Finalizar GIF. Criar imagem 1280x640 do social preview. Instalar OBS Studio. Escrever roteiro do vídeo (8-10 frases).

~5h
Dia 5

Gravar Vídeo + Inserir Assets

Praticar roteiro 2x, gravar na 3ª. Edição básica (cortar início/fim). Inserir screenshots e GIF nos READMEs.

~5h
Dia 6

Texto HN + Product Hunt

Redigir título e comentário do Hacker News. Montar página do Product Hunt (tagline, descrição, screenshots, vídeo).

~5h
Dia 7

Threads X + Posts Reddit + Email

Threads EN e PT-BR para X/Twitter. Posts adaptados para cada subreddit. Template de cold email. One-pager PDF.

~5h
Dia 8

Lista de Contatos

Pesquisar e organizar todos os contatos: Polymarket, Kalshi, jornalistas, newsletters, acadêmicos, influenciadores BR.

~4h
Dia 9

Teste Git Clone + Stress Test

Testar setup do zero seguindo README. Verificar estabilidade do site. Corrigir qualquer problema encontrado.

~4h
Dia 10

Agendar PH + Revisão Final

Agendar Product Hunt. Revisar TODOS os textos (HN, X, Reddit, email). Preparar FAQ. Definir data do Dia D.

~4h
Dia 11

Buffer / Respiro

Dia de folga ou para resolver pendências. Descansar antes do Dia D. Revisar tudo uma última vez com calma.

~2h
Dia 12

DIA DO LANÇAMENTO

Repo público. HN. X. Reddit. Emails. Responder tudo o dia inteiro. Dia mais importante de todos.

~8h
D+1 a D+7

Sustentação + Comunidade

Responder issues/PRs. Follow-up contatos. Postar métricas. Escrever blog post. Avaliar próximos passos.

~1-2h/dia
